# 사용자 흐름

Gate 3. `company-scout` Codex 스킬의 핵심 흐름.

## 주 흐름: 회사 1곳 스카우트
1. 사용자가 Codex에서 스킬 호출 + 회사명 입력 (예: "마이리얼트립").
2. 스킬(SKILL.md 지시)이 5축으로 공개 출처 수집:
   - 회사: 공식사이트·법인정보·뉴스
   - 제품: 공식 제품/기능·기술블로그
   - 고객: 공식 소개·뉴스
   - 문제: 채용 JD·기술블로그 난제·뉴스/IR 과제·앱리뷰(C단서)
   - 관련회사: 투자사·경쟁사·고객사·M&A(관계유형 라벨)
3. 각 출처에 등급(A/B/C) 부여, 확인일 기록.
4. 문제 후보 추출(≥3) — 각 문제·공개URL·Codex 적합성 메모.
5. dossier(마크다운 보고서) 조립: 고정 스키마 + References + "근거 부족" 분리.
   - **각 섹션/파트(회사·제품·고객·문제·관련회사) 끝에 꼬리질문 3개**를 붙인다 — 아직 안 판 곳·검증 필요·확장 방향을 가리키는 질문(예: "자회사 명단은 DART로 확인 가능한가?").
   - **용어 설명**: 전문 용어(예: GDS, OTA, RPE)는 본문에서 링크하고 맨 밑 **용어 설명 섹션**에 한 줄 정의 + 가능하면 출처. 간단·중요한 용어는 본문 괄호로 1줄 인라인.
6. (Should) 출처-완전성 점검: 문제 후보에 URL 없으면 경고.
7. 사용자는 dossier를 받아 예선 플러그인 기획 입력으로 사용.

## 예외/경계 흐름
- 공개 출처가 빈약한 회사 → 해당 축을 "근거 부족"으로 명시(지어내지 않음).
- 출처 충돌(예: 매출 수치 상반) → 양쪽 출처 병기 + "충돌, 재확인 필요" 표기.
- 유튜브/커뮤니티만 있는 항목 → C등급 단서로만, 사실 단정 금지.
- 본선 문제 예고(유튜브)와 겹치는 문제 → 독립 공개 출처로 별도 입증된 것만 채택.
